# 新增渠道指南

## 一、先确认数据来源

更完整的排查方法见 [API 发现手册](api-discovery-playbook.zh-CN.md)。

按以下顺序检查：

1. 页面中的 JSON-LD 或商品 meta。
2. 浏览器 Network 面板里的 REST、GraphQL、tRPC 或搜索接口。
3. 服务端 HTML。
4. 必须执行 JavaScript 时再使用 Playwright。

不要默认从可见文本中猜价格。机器可读字段通常更稳定，也更容易区分当前价、原价和分期价。

## 二、建立 adapter

先生成 adapter 骨架：

```bash
retail-scrape scaffold example-shop-us \
  --display-name "Example Shop US" \
  --country US \
  --with-fixtures
```

它会创建：

```text
src/retail_scrapers/adapters/example_shop_us/
├─ __init__.py
└─ adapter.py
tests/test_example_shop_us_adapter.py
tests/test_example_shop_us_fixtures.py
tests/fixtures/example_shop_us/
├─ catalog.sample.json
└─ price.sample.json
```

实现 `ChannelAdapter` 的两个方法：

- `scrape_catalog`
- `scrape_prices`

然后在 `registry.py` 注册。

## 三、定义完整性条件

渠道如果提供总量或页数，必须交叉校验：

- 每页声明的总量保持一致。
- 最终唯一 SKU 数等于来源总量。
- SKU 或 URL 不跨页重复。
- 筛选后的品牌、年份与返回记录一致。

如果网站没有总量，至少设置非空、重复和成功率检查，并在文档中说明限制。

## 四、添加离线测试

测试不得依赖每天都能访问目标网站。应保存脱敏后的最小 HTML 或 JSON fixture，测试：

- 字段解析。
- 价格与币种。
- 分页和重复检测。
- 空响应与异常响应。

真实网站 smoke test 应使用手动触发的 workflow，避免每个 Pull Request 都访问零售网站。

如果使用了 `--with-fixtures`，请把默认的 `example.com` 合成样本替换为公开来源的最小脱敏样本。fixture 应足够小，方便 PR review。

## 五、提交前检查

```bash
ruff check .
pytest
mypy src/retail_scrapers
```

确认没有提交：

- 抓取结果。
- Cookie、Token 或账号。
- 个人路径和内部域名。
- 用户业务映射或商品主数据。
